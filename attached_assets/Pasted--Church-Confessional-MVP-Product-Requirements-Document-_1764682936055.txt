## 1. Executive Summary

### Product Vision

A platform where congregation members can anonymously share experiences from their churches and receive community validation or pushback‚Äîcreating crowdsourced accountability for pastoral leadership and church culture.

### Problem Statement

Church members who witness or experience misconduct, manipulation, financial impropriety, or toxic culture often have no safe outlet. Internal reporting risks retaliation, social isolation, or gaslighting. There‚Äôs no ‚ÄúGlassdoor for churches.‚Äù

### MVP Goal

Launch a functional, anonymous submission and voting platform that validates the concept and gathers initial user feedback within 4-6 weeks.

## 2. Roundtable Discussion

### üéØ Product Manager ‚Äî Scope & Priorities

**MVP Must-Haves (P0):**

- Anonymous confession/experience submission
- Binary community judgment (Condemn / Absolve)
- Browse/read submissions feed
- Basic category tagging
- Content moderation queue

**Post-MVP (P1):**

- Church/denomination tagging (searchable)
- User accounts (still anonymous posting)
- ‚ÄúMe too‚Äù counter (others with same experience)
- Resource links for abuse/crisis
- Commenting (anonymous)

**Out of Scope for MVP:**

- Pastor/church name search
- Verified church directories
- Direct messaging
- Mobile native apps

**Success Metrics:**

- 100 submissions in first 30 days
- 60% of posts receive 10+ votes
- <5% moderation removal rate (quality signal)

### üìä Product Strategist ‚Äî Positioning & Risk

**Target Users:**

- Current/former church members processing experiences
- People researching churches before joining
- Advocates for church accountability

**Competitive Landscape:**

|Platform              |Gap We Fill                           |
|----------------------|--------------------------------------|
|Church Clarity        |Policy-focused, not experiential      |

|Risk                          |Mitigation                                       |
|------------------------------|-------------------------------------------------|

|Fake/troll posts              |Rate limiting, moderation queue, community voting|
|Trauma dumping without support|Surface crisis resources on sensitive posts      |

**Go-to-Market (MVP):**

- Soft launch in ex-vangelical/church hurt communities (Twitter/X, Reddit, Threads)
- No paid marketing initially
- Gather testimonials for credibility

### ‚ö° Interaction Designer ‚Äî Micro-interactions

**Voting Interaction:**

- Two clear judgment options: **Condemn** / **Absolve**
- Condemn = ‚ÄúThis behavior was wrong‚Äù
- Absolve = ‚ÄúThis behavior was justified or acceptable‚Äù
- Tap/click to vote
- Button fills with color + subtle scale animation (1.05x, 150ms ease-out)
- Vote count increments with number tick animation
- User can change vote (toggle behavior)
- Voting state persists via localStorage (no login)

**Submission Flow:**

- Character counter on textarea (live)
- Submit button disabled until minimum met
- On submit: button ‚Üí spinner ‚Üí checkmark ‚Üí redirect
- Success toast: ‚ÄúYour voice matters. Thank you.‚Äù

**Content Loading:**

- Skeleton cards while loading
- Infinite scroll with ‚ÄúLoad more‚Äù fallback
- Pull-to-refresh on mobile

**Flag Flow:**

- Flag icon ‚Üí confirmation modal ‚Üí ‚ÄúThank you, we‚Äôll review‚Äù ‚Üí icon changes state

**Empty States:**

- No posts in category: ‚ÄúNo experiences shared yet. Be the first.‚Äù
- Error state: ‚ÄúSomething went wrong. Please try again.‚Äù

**API Endpoints (MVP):**

*Public:*

```
GET    /api/submissions         # List (paginated, filterable) ‚Äî excludes admin fields
GET    /api/submissions/:id     # Single submission ‚Äî excludes admin fields
POST   /api/submissions         # Create new (accepts admin-only fields)
POST   /api/submissions/:id/vote    # Cast vote
POST   /api/submissions/:id/flag    # Flag content
GET    /api/categories          # List categories
```

*Admin (protected):*

```
GET    /api/admin/submissions   # Full list INCLUDING church/pastor/location
GET    /api/admin/submissions/:id   # Single with all fields
GET    /api/admin/patterns      # Group by church_name or pastor_name (future)
PATCH  /api/admin/submissions/:id   # Update status (approve/remove)
```

**Key Technical Decisions:**

1. **Anonymity Approach:**
- No accounts required for MVP
- Rate limit by IP (5 submissions/day, 50 votes/day)
- Store hashed IP for vote deduplication (not raw IP)
- localStorage token for client-side vote state
1. **Moderation:**
- Auto-flag threshold: 3+ flags ‚Üí under_review
- Admin dashboard (/admin) protected by env variable password
- Admin view shows: full submission + church name + pastor name + location
- Admin can: approve, remove, or mark as ‚Äúpattern watch‚Äù
- Admin can filter/search by church name or pastor name to identify repeat reports
1. **Performance:**
- Paginate feed (20 posts per load)
- Index database on category, created_at, status
- Cache category counts
1. **Security:**
- Input sanitization (DOMPurify for content)
- Rate limiting (express-rate-limit)
- No PII stored
- HTTPS via Replit default
